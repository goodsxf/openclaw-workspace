# ⚡ 主动汇报规则

## 基本原则
1. **零等待原则**：不让零号猜测工作状态
2. **透明原则**：所有工作状态完全可见
3. **及时原则**：状态变化立即通知

## 汇报触发条件

### 必须汇报的情况
1. ✅ **任务开始** - 收到明确任务指令后
2. 📊 **进度更新** - 每完成一个里程碑
3. ⚠️ **遇到阻碍** - 任何导致任务暂停的问题
4. 🎯 **任务完成** - 任务目标达成
5. ⏰ **超时预警** - 预计完成时间将延迟30分钟以上
6. 🔄 **计划变更** - 任务执行顺序或内容变更

## 汇报格式

### 任务开始汇报
```
[任务开始] TASK-XXX: 任务描述
├── 所属项目: 项目名称
├── 项目分类: 分类名称  
├── 预计完成: XX:XX
└── 看板链接: 工作状态看板.md
```

### 进度更新汇报
```
[进度更新] TASK-XXX: 当前进度 XX%
├── 完成内容: 简要描述
├── 下一步: 下一步计划
└── 状态看板已更新
```

### 问题汇报
```
[问题报告] TASK-XXX: 问题描述
├── 影响: 对任务的影响程度
├── 建议: 建议解决方案
├── 需要: 是否需要协助
└── 预计延迟: XX分钟/小时
```

### 任务完成汇报
```
[任务完成] TASK-XXX: 任务描述
├── 结果: 完成情况总结
├── 用时: 实际用时
├── 产出: 交付物清单
└── 建议: 下一步建议
```

## 看板更新规则

### 实时更新项目
1. **任务状态变化**（进行中→完成、进行中→受阻等）
2. **进度百分比变化**（每完成10%或一个重要节点）
3. **时间信息更新**（开始时间、预计完成时间）
4. **阻碍信息添加/清除**

### 定时更新项目
1. **每日18:00** - 生成工作日报
2. **每周一10:00** - 生成周报
3. **每月1日10:00** - 生成月报

## 项目分类标准

### 一级分类
1. **内部管理** - 系统优化、流程改进、工具开发等
2. **电商运营** - 电商相关项目
3. **产品开发** - 软件开发、产品设计等
4. **生产制造** - 生产相关项目

### 任务编号规则
```
TASK-XXX
├── XXX: 三位数字，从001开始
├── 按创建时间顺序编号
└── 完成后保留编号，不重复使用
```

## 状态标识说明

### 任务状态
- 🟢 **进行中** - 正在执行
- 🟡 **受阻** - 遇到问题，需要协助
- 🔵 **等待** - 等待外部输入或资源
- ⚪ **待开始** - 已规划，未开始
- ✅ **已完成** - 任务完成
- ❌ **已取消** - 任务取消

### 项目状态
- 🟢 **进行中** - 有活跃任务
- 🟡 **规划中** - 正在规划，无活跃任务
- 🔵 **暂停** - 暂时停止
- ⚪ **未开始** - 已立项，未开始
- ✅ **已完成** - 项目完成
- 📊 **维护中** - 项目上线后的维护

## 异常处理流程

### 遇到阻碍时
1. **立即停止** - 停止当前任务执行
2. **分析问题** - 分析问题原因和影响
3. **单独通知** - 立即向零号单独汇报
4. **更新看板** - 在看板中标记为"受阻"
5. **等待指令** - 等待零号指示或自行提出解决方案

### 通信中断时
1. **重试3次** - 尝试重新建立通信
2. **记录日志** - 在本地记录所有操作
3. **安全暂停** - 暂停可能产生影响的敏感操作
4. **恢复后汇报** - 通信恢复后立即汇报期间所有操作

## 质量保证

### 汇报质量要求
1. **准确性** - 信息必须准确无误
2. **及时性** - 状态变化后5分钟内必须汇报
3. **完整性** - 包含所有必要信息
4. **可操作性** - 提供明确的下一步建议

### 看板质量要求
1. **实时性** - 状态变化后立即更新
2. **一致性** - 与汇报内容保持一致
3. **可读性** - 信息清晰，易于理解
4. **可追溯** - 保留历史状态记录
